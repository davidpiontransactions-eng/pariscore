Analyse +/− pour PariScore

✅ Avantages

  • ⚡
    1 ligne de code, résultats en secondes Exploration rapide sans écrire une seule ligne de matplotlib. Idéal pour valider un nouveau dataset (SPS, Elo, archive_matches).
  • 🐍
    Python déjà dans le stack PariScore utilise déjà Python (cron_sps_updater.py). AutoViz s'intègre sans nouveau runtime sur le VPS ou en local.
  • 🔍
    FixDQ() — audit qualité données Détecte outliers, missing values, distributions anormales. Parfait pour auditer les 32 918 records de pariscore.db après chaque ETL.
  • 🎯
    Support variable cible (depVar) Specifier depVar='home_win' génère automatiquement les corrélations de toutes les features vs l'outcome — gold mine pour calibrer le modèle Poisson.
  • 📊
    Output HTML interactif (Bokeh) Les rapports Bokeh peuvent être servie en tant que dashboard interne dev — zero front-end code requis.
  • 📄
    Apache 2.0 — aucune restriction commerciale Utilisable dans PariScore sans contrainte légale, même en SaaS Pro €19/mo.

❌ Inconvénients

  • ⏸️
    Inactivité : dernier update juin 2024 12 mois sans commit majeur. Risque de compatibility drift avec pandas 2.x / numpy 2.x. Utiliser avec version Python/pandas épinglées.
  • 💾
    Dépendances lourdes HoloViews + Bokeh + hvPlot = installation volumineuse sur le VPS OVH Node.js-first. Recommandé en local dev uniquement.
  • 🔴
    Non production-ready Génère des fichiers statiques Python-side. Impossible de remplacer Chart.js/ECharts dans pariscore.html ni les SSE/WebSocket live.
  • ⚽
    Zéro feature sports-specific Pas de courbes Elo, ROI betting, head-to-head, surface clay/grass. Nécessite du post-processing custom pour tout contexte tennis/foot.
  • 📉
    Plafond 150k lignes / 30 colonnes (défaut) OK aujourd'hui (32k records), mais les objets match PariScore ont >30 champs. Nécessite max_cols_analyzed override.
  • ⚙️
    Compatibilité HoloViews fragile Issues documentées avec pandas/numpy récents. Tests obligatoires avant adoption — ne pas assumer que pip install suffit.
Insights
🎯
Meilleur cas d'usage : R&D de nouveaux modèles

Quand tu développes un nouveau signal (ex : intégration xvalue.ai ML scouting, calibration Pinnacle OddsPapi), AutoViz peut générer en 10 secondes les distributions, corrélations et outliers du dataset d'entraînement — sans écrire une seule ligne de matplotlib.

⚠️
Risque principal : maintenance stagnante

12 mois sans update majeur sur un outil qui dépend de HoloViews et Bokeh est un signal d'alarme. La plupart des issues signalent des conflits pandas 2.x. Épingler pandas<2.0 dans le venv AutoViz ou utiliser ydata-profiling (alternative mieux maintenue) pour l'audit de prod.

💡
Pattern recommandé : SQLite → CSV → AutoViz

Export des tables SQLite (archive_matches, player_surface_scores) via Python, puis AutoViz en une ligne. Résultat : rapport HTML Bokeh interactif pour review interne. Aucune modification de server.js requise.

🏆
Alternative supérieure pour cas d'usage PariScore : ydata-profiling

Si l'objectif est l'audit complet de pariscore.db (stats, corrélations, missing, types), ydata-profiling (ex pandas-profiling) offre une profondeur statistique bien supérieure et est activement maintenu en 2026. AutoViz gagne uniquement sur la vitesse brute.

🚀
Valeur réelle : accélère le cycle feature → validation

Pour un dev solo (CTO PariScore), l'économie de temps est réelle lors de l'exploration de nouvelles features (surface clay impact, bet signal calibration). Estimation : 30–60 min de matplotlib code économisées par session EDA. Sur 10 sessions de R&D, gain d'une journée entière.
